What We Build

Five capabilities. Real reliability.

Each capability is a working surface we wire on your stack — contract first, instrumented from day one, hardened against the OWASP API Top 10.

Capability 01

API design & contract-first build

The problem. Endpoints drift from documentation. Each deploy breaks a consumer. Frontend and backend integrate in series instead of in parallel, and the spec is whatever the last engineer remembers.

What we build. OpenAPI 3.1 as the source of truth. We generate client SDKs, mock servers, request validators and contract tests from the spec. Frontend builds against the mock while backend builds against the same contract.

Example use case. A SaaS platform on contract-first OpenAPI shipped 40% faster integration cycles and 35% fewer API defects (Postman 2024 State of the API, cited 2025). Contract-test automation cut integration-test time by up to 60%.

Under the hood. OpenAPI 3.1, Spectral linting, Prism and WireMock mocks, Pact and Schemathesis contract tests, Stoplight or Redoc for docs, GitHub Actions to gate the merge.

Capability 02

Third-party integrations (iPaaS alternative)

The problem. Hand-rolled point-to-point connectors break on every vendor schema change. A single hand-coded integration runs three to six months of engineering, and maintenance eats 30–40% of two-year integration cost.

What we build. A typed integration layer the client owns: domain-modelled adapters, an anti-corruption layer at each vendor boundary, versioned OpenAPI and AsyncAPI contracts, and an event bus for fan-out. No platform licence to renew.

Example use case. A South African mid-market enterprise replaces twelve brittle ERP, CRM and payments connectors with a governed integration layer. Vendor schema changes become localised refactors instead of multi-day fire drills.

Under the hood. Node, Go or Python microservices, Kafka or Redis Streams, OpenAPI plus AsyncAPI, OpenTelemetry traces, Terraform for infrastructure.

Capability 03

Payment gateways (PCI-DSS, idempotency, reconciliation)

The problem. Payment integrations without idempotency keys produce duplicate charges on every network timeout. Without reconciliation, mismatches between gateway, ledger and ERP go undetected until finance catches them in a month-end variance.

What we build. Contract-first integration to PayFast, Peach Payments, Yoco, Stitch, Stripe or Adyen with client-generated V4 UUID idempotency keys, exponential backoff with jitter, signed-webhook verification on the raw body and a reconciliation worker proving the ledger matches the gateway daily.

Example use case. A South African retailer switches PSP from PayFast to Peach Payments with zero duplicate charges. Idempotency-key overhead is under 2ms per request (Stripe's documented baseline). A weekly reconciliation report goes to finance.

Under the hood. PSP SDKs, Postgres ledger with an event_id UNIQUE constraint, BullMQ or Sidekiq workers, signature-verified webhooks per Stripe webhook best practices, OpenTelemetry traces, PCI-DSS 4.0 evidence trail.

Capability 04

Webhook delivery & event-driven

The problem. Synchronous webhook dispatch couples the producer to every consumer's uptime. Without jitter, recovered consumers get hammered by synchronised retry storms, and lost events go undiagnosed until a customer complains.

What we build. Queue-backed dispatcher on SQS, RabbitMQ or Kafka. Exponential backoff with random jitter, dead-letter queue for unprocessable events, HMAC-signed payloads and an event-ID dedup table at the consumer with a UNIQUE constraint.

Example use case. A SaaS platform rebuilds webhook delivery with retry-with-jitter and a DLQ. Synchronised retry spikes drop by over 80% (industry pattern data), lost events approach zero, and double-processing is eliminated at the consumer.

Under the hood. SQS, RabbitMQ or Kafka, HMAC signatures, V4 UUID event IDs, a Postgres dedup table, dead-letter queue with replay tooling, OpenTelemetry per route.

Capability 05

Reliability (circuit breakers, rate limits, observability)

The problem. Partner downtime cascades into your service. Rate-limit misconfiguration produces false 429s on legitimate bursts or unbounded queue depth on the wrong endpoints. P99 incidents are diagnosed by guesswork.

What we build. Resilience4j-style circuit breakers (Closed / Open / Half-Open) around every outbound third-party call, token-bucket rate limits on public endpoints, OpenTelemetry traces with histogram buckets aligned to the SLO target, and correlation IDs propagated end-to-end.

Example use case. A South African mid-market enterprise drops symptom-to-cause time on partner-call incidents from hours of guessing to a single trace. Cascading failures stop at the breaker; tail latency on partner calls becomes survivable.

Under the hood. Resilience4j, Polly or opossum, Redis-backed token-bucket, OpenTelemetry collector, Prometheus, Grafana, Jaeger or Tempo for trace storage.

How We Ship

Production APIs are a different discipline.

Contract-first OpenAPI

The spec is the source of truth. Endpoints, types, errors and pagination live in the OpenAPI file. SDKs, mocks and tests generate from it. No prose-documented APIs and no surprise breakage between frontend and backend.

Contracts before code.

Idempotency keys on every write

Client-generated V4 UUIDs, 24-hour TTL minimum, a deduplication table with a UNIQUE constraint. Stripe's documented pattern. Under 2ms overhead per request and an entire class of double-charge bugs eliminated.

Safe to retry, by design.

Semver versioning & deprecation

Breaking changes ship under a new major version. Deprecated versions stay live for a documented sunset window with a header that warns the consumer. No silent breakage; no version-pinning archaeology.

No surprises, ever.

Rate limits + retry-with-jitter

Token-bucket on inbound traffic (burst-friendly). Exponential backoff with random jitter on outbound retries, capped retry count, dead-letter queue at the tail. Jitter alone removes the synchronised retry spike that crashes a recovering endpoint.

Spikes absorbed, not amplified.

OAuth 2.0 + refresh-token rotation

Short-lived access tokens (15–60 minutes), longer-lived refresh tokens (7–14 days) with rotation and reuse-detection. HS256 or RS256 signing. Refresh tokens in HTTP-only cookies, never localStorage. mTLS at sensitive gateway edges.

Auth that survives audit.

Assessment → Build → Retainer

Three engagement shapes. A one-week Integration Assessment. A six- to twelve-week API Design + Build. A monthly Reliability Retainer with on-call coverage and regulatory reporting support. Pricing scoped after the assessment.

Foundation

Contracts before code. Every endpoint, every release.

Every API we ship is grounded in a written OpenAPI contract — designed, reviewed and versioned alongside the implementation. We treat the contract the way we treat database schema: a first-class artefact that ships through review, gets exercised by tests, and never gets edited blindly in production. It is how we get to "this works the same on every consumer" instead of "this works on the consumer we tested with".

Common Questions

Answers before you book

How long is a typical API engagement?

An Integration Assessment is one week, fixed-fee. An API Design + Build runs six to twelve weeks for a focused surface (a single domain, fifteen to twenty-five endpoints), or three to six months for a multi-domain platform. The baseline: a single hand-coded integration on a regular team runs three to six months. Contract-first compresses that materially.

Can you work with our existing API?

Yes. We start with an audit — OpenAPI spec extraction (or first-time authoring), a contract-test suite, an observability gap analysis and a security review against the OWASP API Security Top 10. We then harden in place: idempotency keys, rate limits, OpenTelemetry, refresh-token rotation, gateway controls. No rewrite required.

Which payment gateways do you support?

South Africa: PayFast, Peach Payments, Yoco, Stitch, Ozow, PayGate and PayShap via BankservAfrica. International: Stripe and Adyen. Direct-bank rails through Stitch: Capitec Pay, Absa Pay and Nedbank Direct EFT. Every payment integration ships with client-generated idempotency keys, signed webhooks, a reconciliation worker and a PCI-DSS 4.0 evidence trail.

What SLAs do you offer?

SLAs match the workload tier. For production APIs we publish an availability SLO (typically 99.9% monthly), a p99 latency SLO and an error-budget policy. For South African financial-services clients we align the reliability layer to SARB Directive 1 of 2024 — a two-hour recovery time objective for critical systems (eight-hour maximum), 24-hour incident notification, 48-hour written report. The Reliability Retainer includes on-call coverage.

How do you handle security and PCI-DSS 4.0?

Baseline: OAuth 2.0 with refresh-token rotation, short-lived access tokens, refresh tokens in HTTP-only cookies, JWT signed with HS256 or RS256, mTLS at sensitive edges. For payments: PCI-DSS 4.0 Req 6.5.6 enforced at the gateway (BOLA mitigation, broken function-level authorisation, full request/response logging) and Req 6.3.2 component inventory. POPIA: data-subject rights are wired into the API, and incident reporting fires as soon as reasonably certain, per the 2025 amendment regulations.

Can we self-host the API platform?

Yes. The default architecture is portable — containerised services, infrastructure-as-code via Terraform, no proprietary platform licence to renew. Deployment targets include your AWS, Azure or GCP account, on-premises Kubernetes, or a South African hosted environment when SARB or banking-grade controls apply.
